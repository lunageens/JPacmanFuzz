Jpacman Fuzzing report - Overview and conclusions

This report presents the key findings and lessons learned from the recent fuzzing exercise conducted on the Pacman application. The objective of this exercise was to assess the robustness and security of the Pacman application through extensive input testing. By analyzing the results, we gained valuable insights into potential vulnerabilities and their root causes. This report aims to provide a comprehensive overview of the lessons learned and recommendations for enhancing the security and reliability of the Pacman application.

Fuzz 1: Manual random testing with binary files and correct action sequence

Context

In this stage, the maps are binary with a normal maximum size. The bytes are chosen randomly. The name of the map file is normal. We will randomly choose bytes with the RandomBinaryMapGenerator.generateRandomMap() method. The action sequence will exist out of 5 characters (randomly chosen out the valid options E, S, U, D, Q, W, L and R). It will have a normal name.

Configurations

Results

The results are stored in the fuzz1_binary directory. Running the Fuzz class with the right configurations would give u something like shown in the above figure. Open the binary files in a binary editor like HxD to see for yourself. We can see that if the binary file has multiple lines, we get an Error reading file by readAllLines. If the binary file has one line, the program JPacman starts to read the line. If the line contains characters that are not valid, it will give another error message. We know from the given assignment text, that a text file can have multiple lines in it and still be read.

This would give us the following informal pseudocode: 

if(file has multiple lines)
   if(file is not a text file)
        reject(Error reading file by readAllLines)
if(file has one line)
    if(file is not a text file)
        if(file is binary file)
            if(line is not a a valid character)
                reject(unknown character)
            if(line is valid character)
                ?
        ?
     ?
if(file has no lines)
    if(file is a binary file)
        reject(no lines)


Fuzz 2: File content ~ Known ASCII characters

Context

If I want to know if one-line binary files can be valid, I first need to know what the valid characters are. Let's make a text map that always includes one of the 128 characters of the ASCII range. We do this by using the method RandomTextMapGenerator.generateCustomTextMap(String mapLine) . The name of the map file is normal. The action sequence will exist out of 3 valid characters in a valid sequence: Start (S) - Wait (W) - Exit (E). The name of the string will be normal.

Results

The results are stored in the directory fuzz2_knownCharacters . A summary of the results is shown in the above figure. If it is a valid file and not an empty one, look at the characters inside. If they are all valid character, look at the presence of all the needed characters. Note also that we did not test for other alphabets or characters outside the ASCII range such as emoji's.

This would give us the following informal pseudocode: 

if (the file is not valid)
    reject(Error reading file by readAllLines)
if(the map does not contain valid characters (M, W, F, P))
    reject(unknown character)
if(the map does not contain character P)
    reject(no player is set)
if(the map does not contain character F)
    reject(no food is set)

Fuzz 3: Valid (binary) file types

Context

We still use all valid action sequence strings, since we are focussing on the map file first.

First, we want to see if there is a difference between the allowed lines for each type.

  • We want to know if we can have a binary file type of one line that is valid. Therefore, test multiple and single character file type encoding with valid string. Those should get error no player of food instead of error reading file by readAllLines.
  • If one-line binary files are valid, are there other file types that can only have one-line? Let's add some single-lined .map, .dat, .xml, .html, .xlsx, .pdf, and .csv files. Use Excel and IntelJI Idea to create those files.
  • Are multiple line text files valid? Are there other file types that can have multiple lines? Let's add some multi-lined .map, .dat, .xml, .html, .xlsx, .pdf, and .csv files. Use Excel and IntelJI Idea to create those files.

Second, are there other restrictions to file types?

  • File compression: Let's add a .txt file, but modify it to a compressed .zip file. Use your operating system.
  • File permission: Let's add a .map file with no , read-only and write-only permission. Try to input a hidden file. Use your operating system.
  • File encryption: Let's add an encrypting algorithm, such as AES, with a password. Use tools such as Veracrypt to create large encrypted files.
  • File corruption: Add a file with an invalid extension (for example, .corrupted ) or change the encoding of a text file from UTF-8 to another one (for example, UTF-16). Use programs like Notepad++ to do so.
  • File size: Let's add an empty and a massive .txt file. Use sites such as dummytextgenerator.com or do this via the command line, as explained here for windows operating systems.
  • File name: Let's add an emtpy name. Let's add special characters (space, @,or characters from mixed alphabets) in the name of the map file.


Configurations

If u want to use my custom_maps directory, copy the file in the custom_maps_inputCopy_3 directory of the fuzz3_filetypes directory. This is to avoid that the program will move them while executing the fuzzer, and certain files will be lost during that process. If u copy-paste the files to maps, make sure to reconfigure permissions files. Files lose their permission while copying. U might notice that you cannot copy the file without any permissions. Since all permissions files have the same content, u can copy the read-only or write-only file twice and reconfigure one copy to no permissions. Make sure to run with none of the files open, otherwise you will receive an error because the application cannot move open files.

Results

The results are stored in the directory fuzz3_filetypes. We are going to have to put the file without permissions out of the list because it gives an error before even reaching the fuzzer.



Let's look at the other files. For the first time, the program accepted some files!

  • We know from the assignment that the program excepts single and multi-line text files. The program accepted multi-line text files, even if they had a file name that included other alphabets, spaces or special characters or if they were hidden. The program rejected the massive text files, but only because it had unknown characters in it.
  • The program accepted the multiple map files, even if they were only read-only. The program rejected the single-lined map files, but only because they have missing essential characters (food or player).



Some rejected file types were:

  • Error message "Error reading file by ReadAllLines": compressed, encoded or encrypted files, write-only files, all pdf or Excel files, and multi-line binary files.
  • Error message "Unknown character": all .corrupted, .csv, .html and .xml files, and multi-line binary or .dat files that have multiple lines. Note that those files were tested on being single or multiple line, and did not contain any unknown characters.
  • Error message "No lines": One cannot input an empty file.


This would give us the following informal pseudocode: 

 if(file has not any permission)
    cannot do input in terminal
 if(file is compressed,not encoded with UTF-8,  encrypted, write-only, or from an invalid data type (pdf, excel or multi-line binary))
    reject(Error reading file by ReadAllLines)
 if(file is .csv, .html, .xml, .corrupted or multi-line .dat) || (the map does not contain valid characters (M, W, F, P))
    reject(unknown character)
 if(file is empty)
    reject(No lines)
 if(the map does not contain character P)
    reject(no player is set)
 if(the map does not contain character F)
    reject(no food is set)
 else
    accept

Results

The logs from running the Fuzzer with customMapsNr 4, 5 and 6 are stored in the directory fuzz4_validCharacterMaps .

As seen before, the program will first see if there is any content at all. If there is not it will give the no lines error message. If there are lines that contain content, but the first line is empty, the error message no columns will appear. If the first line has content, but is has two Ps, then the error message multiple players will appear. After that, it will check the line width of the next line. If the map is not rectangular, it will give an error message width mismatch. If the second line is equally long, but contains multiple ps, the error message multiple players will appear. For each line that follows, it will first check the width and then check if there is not more than one player. After reading the whole file, it will check if the map has a player at all. If not, the error message no player is set will appear. If there is just one player, but there is no food, the program rejects again: no food will be seen on the console. If all these conditions are fulfilled, the program accepts the map. Note that maps without any monsters also pass. In the context of the Pacman application, this means that a map is allowed where the player cannot die.
