Jpacman Fuzzing report - Overview and conclusions

This report presents the key findings and lessons learned from the recent fuzzing exercise conducted on the Pacman application. The objective of this exercise was to assess the robustness and security of the Pacman application through extensive input testing. By analyzing the results, we gained valuable insights into potential vulnerabilities and their root causes. This report aims to provide a comprehensive overview of the lessons learned and recommendations for enhancing the security and reliability of the Pacman application.

Fuzz 3: Valid (binary) file types

Context

We still use all valid action sequence strings, since we are focussing on the map file first.

First, we want to see if there is a difference between the allowed lines for each type.

  • We want to know if we can have a binary file type of one line that is valid. Therefore, test multiple and single character file type encoding with valid string. Those should get error no player of food instead of error reading file by readAllLines.
  • If one-line binary files are valid, are there other file types that can only have one-line? Let's add some single-lined .map, .dat, .xml, .html, .xlsx, .pdf, and .csv files. Use Excel and IntelJI Idea to create those files.
  • Are multiple line text files valid? Are there other file types that can have multiple lines? Let's add some multi-lined .map, .dat, .xml, .html, .xlsx, .pdf, and .csv files. Use Excel and IntelJI Idea to create those files.

Second, are there other restrictions to file types?

  • File compression: Let's add a .txt file, but modify it to a compressed .zip file. Use your operating system.
  • File permission: Let's add a .map file with no , read-only and write-only permission. Try to input a hidden file. Use your operating system.
  • File encryption: Let's add an encrypting algorithm, such as AES, with a password. Use tools such as Veracrypt to create large encrypted files.
  • File corruption: Add a file with an invalid extension (for example, .corrupted ) or change the encoding of a text file from UTF-8 to another one (for example, UTF-16). Use programs like Notepad++ to do so.
  • File size: Let's add an empty and a massive .txt file. Use sites such as dummytextgenerator.com or do this via the command line, as explained here for windows operating systems.
  • File name: Let's add an emtpy name. Let's add special characters (space, @,or characters from mixed alphabets) in the name of the map file.

Results

The logs from running the Fuzzer with customMapsNr 4, 5 and 6 are stored in the directory fuzz4_validCharacterMaps .

In the fuzz attempt with customMaps 4, almost 60% had more than one player because there were two or more players on the first line. If that was not the case, there was another mistake such as width Mismatch or no lines, ensuring that no map file was accepted by the application. To diversify the output messages, we tried fuzzing with customMaps 5, which only generates random square maps with valid characters. However, all map files were still rejected. 94 percent of them were rejected because there was more than one player. This is why we created custom maps to observe which output messages contribute to rejection and in which order.



For customMaps 6, the following important output messages were observed:

  • No lines: If all rows are empty, the file is rejected (e.g., FileEmpty.txt).
  • No columns: If the first line is empty but other rows have characters in them, the output message "No columns" will be received (e.g., FirstLineEmpty.txt).
  • More than one player: If the first line is not empty but contains more than one character P in the total map file, the output message "More than one player" will appear (e.g., FirstLinePlayers.txt and LinePlayers.txt ).
  • Widths Mismatch: If any line is shorter or longer than the first line (even lines without characters), the output message "Widths Mismatch" is shown (e.g., LineEmpty.txt , LineTooLong.txt , and LineTooShort.txt ).
  • No player is set: If the total map file does not contain any character P, the output message "No player is set" is received (e.g., NoPlayer.txt ).
  • No food: If the total map file does not contain any character F, the output message "No food" is received (e.g., NoFood.txt ).

Some map files also got accepted:

  • NoMonster.txt : Maps without any monsters pass, meaning a map is allowed where the player cannot die.
  • MultipleFood.txt and MultipleMonsters.txt : Multiple foods or monsters in the map are not a problem.
  • Rectangular.txt : Rectangular maps, unlike our custom square maps, are also accepted if there are no other rejected attributes.


One might wonder how to determine the hierarchy of these output messages. An indication could be the occurrence percentage of the output message. For example, "Widths Mismatch" was the output message for 37% of all map files, suggesting its higher position in the hierarchy. To be sure, we combined as many error messages as possible without unnecessary duplication. Each output message was tested against all output messages below it.

Firstly, there are some attributes of the first line that are checked:

  • No lines: If the file is empty, it is not possible to receive an error message about its content. Therefore, "No lines" is the highest in the hierarchy.
  • No columns: It is not possible to have the first line empty and have two players in the first line. We did test the case where two players occurred in another line than the first line ( FirstLineEmpty_LinePlayers.txt ). To test for width Mismatch, we made the first line empty and the third line either too short or too long ( FirstLineEmpty_LineTooShort.txt and FirstLineEmpty_LineTooLong.txt ). We also tested map files where the first line was empty but had no players, food, or monsters ( FirstLineEmpty_NoPlayer.txt , FirstLineEmpty_NoFood.txt and FirstLineEmpty_NoMonster.txt ). Except for completely empty map files, invalid map file types, and unknown characters, all map files with an empty first line received the output message "No columns," even if there were other issues with the map file's content in subsequent lines.
  • More than one player (in the first line): If the first line is not empty but contains the character "P" more than once, the output message "More than one player" will always be displayed. This is true even if other lines contain more "P"s ( FirstLinePlayers_LinePlayers.txt ), are shorter ( FirstLinePlayers_LineTooShort.txt ), or longer ( FirstLinePlayers_LineTooLong.txt ). It also applies when the entire map file does not contain any "F" characters ( FirstLinePlayers_NoFood.txt ) or "M" characters ( FirstLinePlayers_NoMonster.txt ). Logically, the combination of the first line having more than one player and the total map file having no players is not applicable.



Next, each line is tested separately in the following order:

  • Widths Mismatch: A line can be either longer or shorter than the first line but never both at the same time. If a line is too short or too long, the output message "Widths Mismatch" will be displayed, even if the total map file does not contain any food, players, or monsters ( LineTooLong_NoFood.txt , LineTooShort_NoFood.txt , LineTooLong_NoPlayer.txt , LineTooShort_NoPlayer.txt , LineTooLong_NoMonster.txt , and LineTooShort_NoMonster.txt ). The question of the combination of "Widths Mismatch" and "More than one player" follows. We have already seen that when there are two players in the first line, the output message "More than one player" is shown. However, what happens if a line is too short or too long, and that same line brings the total number of players to more than one? This can occur in the following ways:
    • The line is too short and has more than one player. Both player characters are before the end of the first line ( LineTooShort_LinePlayers.txt ).
    • The line is too long and has more than one player. Both player characters can be found completely before the end of the first line ( LineTooLong_LinePlayers_Before.txt ), on the column of the end of the first line ( LineTooLong_LinePlayers_On.txt ), or completely after the end of the first line ( LineTooLong_LinePlayers_After.txt ).
    • The line is too short and only contains one player but brings the total players to more than one. That one player character is placed fully before the end of the first line ( LineTooShort_TotalPlayers_Before.txt ).
    • The line is too long and only contains one player but brings the total players to more than one. That one player character can be placed before the end of the first line ( LineTooLong_TotalPlayers_Before.txt ), on the column of the end of the first line ( LineTooLong_TotalPlayers_On.txt ), or after the end of the first line ( LineTooLong_TotalPlayers_After.txt ).
    In these cases, the output message "Widths Mismatch" is assigned, indicating that the length of the line is calculated and compared to the length of the first line before the line is read in, and therefore the position of the extra players does not make a difference (in contradiction to testing each separate character against both criteria). All the mentioned map files receive the output message "Widths Mismatch", confirming this conclusion.
  • More than one player: If there is no width Mismatch in the line, the line is read. Only when all lines are read can we be sure that the map file does not contain any food, players, or monsters. Therefore, a map file containing two players in any line but no food ( LinePlayers_NoFood.txt ) or monsters ( LinePlayers_NoMonster.txt ) will always display the "More than one player" output message. As mentioned before, the combination of the output messages "More than one player" and "No player" does not make any sense.



Regarding this, there are some additional notes:

  • To demonstrate the importance of line order, we have added a file where one line has two players and the line after that is shorter than the first line ( LinePlayers_LineTooShort.txt ). Even though the "Widths Mismatch" output message is higher in the hierarchy than "More than one player," it's important to keep in mind that these attributes are checked per line. If there is a line with the correct width but two players, and no incorrect attributes such as incorrect widths have occurred before that, it will result in the "More than one player" output message, even if following lines have incorrect widths.
  • To demonstrate that it does not matter if a line is too short or too long, observe that the map file with both a shorter and a longer line ( LineTooShort_LineTooLong.txt ) will also result in the "Widths Mismatch" output message.


In third place, the occurrence of food and players is checked after reading all the lines.

  • No player: If there are no players or no food, the output message "No player" is displayed ( NoPlayer_NoFood.txt ).
  • In combinations with no monsters, the other output messages will always be shown, as only having no monsters does not result in rejection by the application ( NoPlayer_NoMonster.txt and NoFood_NoMonster.txt ).

Finally, an important question is where this group of error messages fits into the existing pseudocode.

  • EmptyFile_UnknownFileType.pdf : A PDF file with no content is rejected first because it is an unknown file type, resulting in the output message "Error reading file by ReadAllLines," instead of being rejected due to lack of content in the map file, which would trigger the "No lines" error message.
  • FirstCharacterEmpty.txt : If it is not the first line that is empty but only the first character, the line is still read. The application detects a space and displays the "Unknown Character" output message.
  • RandomCharacterEmpty.txt : Similarly, if the first line contains a space in a different column of the first line, it also results in the "Unknown Character" output message.


In conclusion, the program first checks if there is any content at all. If there isn't, it displays the "No lines" error message. If there are lines with content but the first line is empty, the error message "No columns" is displayed. If the first line has content but contains two "P" characters, the error message "Multiple players" is displayed. Then, it checks the line width of the next line. If the map is not rectangular, it displays the "Widths Mismatch" error message. If the second line is equally long but contains multiple "P" characters, the "Multiple players" error message appears. For each subsequent line, it first checks the width and then checks if there is no more than one player. After reading the entire file, it checks if the map has at least one player. If not, the "No player is set" error message is displayed. If there is only one player but no food, the program rejects the map, showing the "No food" error message. If all these conditions are met, the program accepts the map. It's important to note that maps without any monsters also pass. In the context of the Pacman application, this means that a map is allowed where the player cannot die.


This would give us the following informal pseudocode: 

if(file has not any permission)
    cannot do input in terminal
 if(file is compressed,not encoded with UTF-8,  encrypted, write-only, or from an invalid data type (pdf, excel or multi-line binary))
    reject(Error reading file by ReadAllLines)
 if(file is .csv, .html, .xml, .corrupted or multi-line .dat) || (the map does not contain valid characters (M, W, F, P))
    reject(unknown character)
 if(file is empty)
    reject(No lines)
if(first line is empty)
	reject(No columns)
if(first line brings the total count of P characters to more than one)
	reject(More than one player)
for each (line : lines in the map file)
	if(line is shorter than first line)
	if(line is longer than first line)
		reject(Widths Mismatch)
	if(line brings the total count of P characters to more than one)
		reject(More than one player)
 if(the map does not contain character P)
    reject(No player is set)
 if(the map does not contain character F)
    reject(No food)
 else
    accept

Fuzz 5: Valid action sequences

Context

Using customMaps 7 and a correct action sequence string will always result in acceptation. It is now time to look at the action sequences. We know from the explanation that the only valid characters are the following:

  • E: Exit the game.
  • Q: Quit the game.
  • S: Start the game.
  • W: Pause the game for fifty seconds.
  • U: Move the player once cell up.
  • D: Move the player once cell down.
  • R: Move the player once cell to the right.
  • L: Move the player once cell to the left.

Limitations and Results

The results are stored in the last fuzzlessons directory. The causes of these above-mentioned crashes remains unclear. One could also test mixes of incorrect maps and sequences, or action sequences that are very long or empty. On top of that, one could apply mutational fuzzing on the JPacman application.


Currently, the following pseudocode is our conclusion: 

if(file has not any permission)
    cannot do input in terminal
 if(file is compressed,not encoded with UTF-8,  encrypted, write-only, or from an invalid data type (pdf, excel or multi-line binary))
    reject(Error reading file by ReadAllLines)

    reject(No lines)

    reject(No columns)

        reject(Widths Mismatch)
